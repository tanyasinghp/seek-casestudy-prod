Why this matters now

Seek achieved something genuinely remarkable: it became the first AI agent to complete Salesforce Trailhead Admin Beginner and Intermediate courses, validating UI-first automation at the level of a certified Salesforce administrator. That's not a demo milestone. That's a proof-of-capability that changes what enterprise buyers believe is possible.

The market is at an inflection point. A good percentage of new enterprise buyers in H2 2025 have adopted AI Agents as part of their rollout. The window to establish Seek as the default enterprise agentic layer, before Microsoft Copilot, Salesforce Agentforce, and ServiceNow Now Assist consolidate is roughly 12–18 months. The product decisions made in this window are permanent.
What's validated
UI-first automation that bypasses API limits. Multi-step task completion in live enterprise systems. Human-agent handoff on the same UI surface. Salesforce admin-level capability in controlled testing.
What's still open
Accuracy at scale across real, messy enterprise data. Token cost per task at enterprise volume. Trust and governance layer for production deployment. The next application beyond Salesforce.

Trust & governance layer

Trust is a repeatedly surfaced keyword when considering products like Seek. Enterprise users are ultimately delegating actions on live systems, making adoption dependent not just on capability, but on confidence. Trust, in this context, feels less like an outcome and more like a product feature that needs to be intentionally designed.

The action classification model

Not all Seek actions carry equal risk, there's a need to classify actions based on their potential impact before determining the appropriate level of autonomy.

Auto-execute
Low-risk, easily reversible actions such as read-only queries or simple configuration updates. Requiring confirmation for these tasks may add friction without meaningfully improving safety.
Preview + confirm
Medium-risk actions with broader implications, such as modifying workflows or page layouts. A diff-style preview allows users to review the proposed changes before execution.
Explicit approval
High-risk actions involving access controls, permissions, billing, or other changes with significant downstream consequences. These actions warrant a higher degree of oversight, with approvals and auditability built into the experience.
Governance requirements for enterprise buyers
Audit trail
Every Seek action should be traceable, capturing who initiated it, what was executed, and what changed.
Rollback design
The ability to reverse actions should be considered upfront, with clear expectations around what can and cannot be undone.
Explainability
Users and compliance teams should be able to understand why Seek took an action, not just what action was taken.
Scope limits
Some actions may warrant hard product-level boundaries rather than relying solely on model judgment.

Where tokens get wasted in agentic systems
Full-context resend Resending the entire execution history at every step causes token usage to scale with task complexity. One question worth exploring is how much context is actually needed for reliable execution.
Screenshot overload Processing full-page screenshots when only a few UI elements matter introduces avoidable overhead. More selective context may offer a better balance between accuracy and efficiency.
Retry storms Retrying failures with the same model can quickly inflate costs. Escalation strategies that adapt based on failure patterns may prove more effective.
Frontier model defaults Simpler tasks may not justify frontier models. Routing requests based on task complexity could improve unit economics without degrading the experience users perceive.
PM perspective for cost: What stood out to me was that token efficiency is as much a product consideration as it is an engineering one. Choices around context, retries, and model selection directly influence unit economics, making cost governance essential for sustainable enterprise adoption.

Optional Technical Deep Dives

Context Engineering Trade-offs

One of the more interesting trade-offs I found while thinking through Seek's architecture was around context engineering. The amount of context supplied to an agent directly influences both action quality and operating cost. Passing the entire execution history and UI state through every reasoning cycle maximizes completeness, but it also causes token consumption to scale with task complexity.

This creates a natural tension. More context can improve grounding and reduce execution errors, whereas less context reduces latency and operating costs but may increase the risk of incorrect actions. There is unlikely to be a single "correct" answer that applies across all workflows.

My instinct would be to optimize for the minimum context required to complete a task reliably, rather than maximizing context by default. This becomes increasingly important as token costs compound across high-volume enterprise workflows.

For repetitive tasks, approaches such as page-indexed retrieval and reusable prompt templates may preserve grounding while avoiding the need to repeatedly pass the full interaction history to the model. This raises an interesting question: should every task be treated identically, or can different categories of workflows operate with different context strategies?

Ultimately, the challenge is not to maximize context at any cost, but to balance:

  • Accuracy
  • Latency
  • Token expenditure
  • User trust

in a way that remains sustainable at enterprise scale. The objective is to identify the minimum context necessary for Seek to execute tasks confidently, consistently, and economically.

Trust & Accountability

Trust Checkpoint: Preserving Accountability

What stood out to me was that the trust checkpoint is not primarily about preventing mistakes—it's about preserving accountability. A Salesforce administrator is accountable by role, risk-calibrated through experience, operating under time pressure, and often subject to audit requirements. The way Seek handles trust directly influences whether users are willing to delegate meaningful work to it.

Three distinct failure modes appear particularly important to account for:

  • Silent wrong action: Seek executes successfully but performs an unintended action.
  • Correct action, wrong scope: The intended operation is applied to the wrong records, users, or environments.
  • Partial completion: A multi-step workflow completes incompletely, leaving the system in an inconsistent state.

One approach I found compelling was to think about trust as a layered mechanism rather than a single approval prompt:

Intent confirmation: Validate the user's objective before execution begins.

Execution transparency: Provide visibility into what Seek is doing while a task is in progress.

Risk-based intervention: Differentiate between actions that can auto-execute, require preview, or need explicit approval.

Outcome verification: Confirm what Seek was able to validate after execution and surface any residual uncertainty.

Another question I would seek to validate is how accountability should evolve over time. Should trust increase as users build confidence through successful interactions? Conversely, what events should reset that trust and reintroduce safeguards? These decisions may ultimately influence adoption just as much as the underlying model performance.

Cost-Aware Routing

Technical Deep Dive: Cost-Aware Model Routing

One hypothesis I found compelling was that Seek should not default to its most capable models. Different classes of tasks demand different levels of reasoning, making routing an important lever for balancing capability, latency, and cost.

Suggested Routing Approach

Task Type Example Suggested Tier
Read-only queries "Show all validation rules" Lightweight models
Single-step configuration Field rename, picklist update Mid-tier models
Multi-step configuration Validation rules, flows Frontier models
High-risk actions Permissions, sharing rules, bulk operations Frontier models with explicit approval

Retry Philosophy

Failures should not trigger unbounded retries. One approach worth validating would be to progressively enrich context, escalate model capability when appropriate, and surface unresolved cases to the user once the expected value of additional retries diminishes.

Context Controls

Potential cost controls include:

  • Summarizing completed steps rather than carrying full histories forward,
  • Restricting visual inputs to relevant screen regions rather than processing entire screens,
  • Anchoring execution against an approved task plan instead of repeatedly reconstructing intent.
What this highlighted for me is that model selection cannot be treated as an implementation detail. Decisions around routing directly influence customer experience, operating margins, and ultimately the economic viability of enterprise-scale adoption.
